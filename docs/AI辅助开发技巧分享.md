# AI 辅助开发技巧分享

> 以 `NumberUpArgs` 多合一打印需求为例，分享如何高效使用 AI Agent 完成需求开发

---

## 1. 核心心法：把 AI 当作你的同事

**关键认知**：AI 不是搜索引擎，不是代码生成器，而是一个**能理解上下文、能推理、能写代码的助手**。你跟它沟通的方式，决定了它的输出质量。

> 把 AI 当作一个刚入职的同事：聪明、速度快、但不了解你的项目和业务。你需要把背景交代清楚，它才能帮你做得准确。

---

## 2. 需求分析阶段：如何把诉求准确告诉 AI

### 2.1 常见误区

```
❌ 错误示范：
帮我写一个 NumberUpArgs 结构体

✅ 正确示范：
我需要在 frameworks/ohprint/include/ohprint.h 中新增一个 Print_NumberUpArgs 结构体，
包含四个参数：numberUp（每页打印页数）、numberUpLayout（布局方向）、mirror（镜像打印）、
pageBorder（边框样式）。
参考同文件中现有的 Print_PrintTask 结构体风格，使用 typedef struct 语法，
遵循项目现有的命名规范（Print 前缀、UpperCamelCase 枚举值）。
```

**区别在哪里**：
| 要素 | ❌ 错误 | ✅ 正确 |
|------|--------|--------|
| 背景 | 没交代 | 说明了所在文件 |
| 参考 | 没提供 | 指明了参考对象 |
| 规范 | 没指定 | 明确提出命名规范 |
| 语法 | 没要求 | 指明了 typedef struct |

### 2.2 提供参考代码/文件

AI 能理解的最快方式是**看代码**，不是听描述。

**推荐做法**：
- 找到项目中类似的实现，告诉 AI："参考 `xxx.h` 中的 `XXX` 结构体风格"
- 把现有的枚举/结构体直接贴给 AI 看
- 告诉 AI 哪些是必须保持一致的，哪些是可以创新的

**案例（NumberUpArgs）**：
```
在开始之前，我让 AI 看了一段现有的结构体定义：
"参考 Print_PrintTask 的结构体风格定义 Print_NumberUpArgs，
包括注释格式、成员排列顺序、typedef 语法"
```

### 2.3 多角度验证理解

描述完需求后，让 AI 复述一遍它的理解：

```
你：帮我写一个多合一打印的参数结构体，需要支持 1/2/4/6/9/16 页每张纸，
以及布局方向、镜像打印、边框样式。

AI：[复述了自己的理解]

你：基本正确，但 numberUp 只接受我列出的那几个枚举值，不接受其他数值，
其他参数的枚举值范围我后续会告诉你。
```

## 3. 代码实现阶段：如何让 AI 写出好代码

### 3.1 分步骤进行，不要一次性大段生成

**推荐的工作流**：

```
Step 1: 让 AI 定义数据结构（枚举 + 结构体）→ 你检查结构是否正确
Step 2: 让 AI 写参数校验逻辑 → 你检查边界条件
Step 3: 让 AI 写 CUPS 参数映射 → 你检查映射是否完整
Step 4: 让 AI 生成测试用例 → 你验证覆盖度
```

**为什么这样做**：
- 大段代码一次性生成，错误难以定位
- 每步都有检查点，出问题快速回滚
- AI 看到前一步的结果，后续生成更准确

### 3.2 指明代码规范

**项目规范要点，提前告知 AI**：

| 规范项 | 项目要求 |
|--------|----------|
| 头文件注释 | `@since 26` 版本标记 |
| 枚举命名 | UpperCamelCase，如 `PRINT_NUMBER_UP_2_PAGES` |
| 成员注释 | `/** 注释内容 */` 格式 |
| 头文件保护 | `#ifndef __ Print_NumberUpArgs` 格式 |
| 缩进 | 4 空格 |
| 错误处理 | 校验后设置默认值而非直接返回错误 |

**示例 prompt**：
```
按照以下规范生成代码：
1. 头文件添加 @since 26 注释
2. 枚举值用 UpperCamelCase，加 PRINT_ 前缀
3. 成员变量用 lowerCamelCase
4. 所有 public 成员必须有 JSDoc 风格注释
5. 校验失败时使用项目默认值，不打印错误日志
```

### 3.3 让 AI 解释它写的代码

每段重要逻辑完成后，问 AI：
- "这段代码的逻辑是什么？"
- "有没有边界情况没处理？"
- "如果输入是 0 或负数会怎样？"

**目的**：确保你理解 AI 写的代码，同时帮助 AI 发现漏洞。

## 4. 调试阶段：如何用 AI 定位问题

### 4.1 提供完整的上下文

**❌ 差劲的报bug方式**：
```
AI：代码报错了
```

**✅ 正确的报bug方式**：
```
AI：编译时报错：
/path/to/file.cpp:123: error: cannot convert 'uint32_t' to 'PrintNumberUp'
完整代码如下：
[粘贴相关代码片段]
上下文：这是 Print_NumberUpArgs 结构体的赋值逻辑，
numberUp 成员类型是 PrintNumberUp 枚举，
我尝试用 static_cast 转换，但报错了。
```

### 4.2 让 AI 分析错误原因

提供完整错误信息后，问 AI：
- "这个编译错误的原因是什么？"
- "应该如何修复？"
- "有没有类似问题的历史案例？"

### 4.3 让 AI 生成修复方案

确认 AI 的分析正确后，让 AI 给出修复代码：

```
AI：请给出修复后的代码片段，只修改必要的部分。
不要大段重写，只展示 diff。
```

## 5. 注意事项：这些坑不要踩

### 5.1 不要直接粘贴敏感信息

```
❌ 禁止：
帮我看看这段打印机的配置有什么问题：printerUri=192.168.xx.xx, token=abc123

✅ 可以：
帮我看看这个 URI 解析逻辑是否有问题：printerUri 是一个 JSON 字符串字段，
可能的格式是 xxx://xxx/printers/xxx
```

### 5.2 不要让 AI 做架构决策

```
❌ 不好：
这个需求应该用什么架构？前端还是后端实现？

✅ 好：
这个需求计划在前端实现，需要你帮我实现 Print_NumberUpArgs 到 CUPS 参数的映射逻辑。
```

### 5.3 不要跳过人工 review

AI 写的代码必须经过人工 review 才能提交。
常见的 review 重点：
- 参数校验逻辑是否完整
- 枚举值是否覆盖所有情况
- 是否引入新的编译警告
- 是否符合项目代码风格

### 5.4 不要忽视 CUPS 参数兼容性

AI 可能生成 `numberUp=3` 这样的值，但 CUPS 的 number-up 只支持 1/2/4/6/9/16。
**你必须告诉 AI 这个约束**，否则生成的代码无法正常工作。

| 安全/权限相关代码 | | ❌ 需要人工复核 |

## 7. 代码规范要点（结合 NumberUpArgs 案例）

### 7.1 枚举定义规范

```cpp
// ✅ 正确：枚举值有清晰前缀，注释说明含义
typedef enum {
    PRINT_NUMBER_UP_1_PAGE = 1,   // 单页打印
    PRINT_NUMBER_UP_2_PAGES = 2,  // 2合1
    PRINT_NUMBER_UP_4_PAGES = 4,  // 4合1
    // ...
} PrintNumberUp;

// ❌ 错误：枚举值没有前缀，难以区分来源
typedef enum {
    ONE_PAGE,
    TWO_PAGES,
    // ...
} NumberUp;
```

### 7.2 结构体注释规范

```cpp
// ✅ 正确：每个成员都有说明，且说明与 CUPS 参数对应
typedef struct {
    /** 每张纸打印的页数（CUPS number-up 核心参数） */
    PrintNumberUp numberUp;
    /** 多页布局排列方向（仅在 numberUp > 1 时生效） */
    PrintNumberUpLayout numberUpLayout;
    // ...
} Print_NumberUpArgs;

// ❌ 错误：缺少注释或注释不清晰
typedef struct {
    PrintNumberUp numberUp;
    PrintNumberUpLayout numberUpLayout;
    // ...
} Print_NumberUpArgs;
```

### 7.3 参数校验规范

```cpp
// ✅ 正确：校验合法值，超出范围使用默认值
uint32_t numberUpValue = static_cast<uint32_t>(numberUpArgs.numberUp);
if (numberUpValue == PRINT_NUMBER_UP_1_PAGE ||
    numberUpValue == PRINT_NUMBER_UP_2_PAGES ||
    // ...只接受预定义的值
    jsonOptions["numberUp"] = numberUpValue;

// ❌ 错误：直接使用枚举值，没有校验边界
jsonOptions["numberUp"] = numberUpArgs.numberUp;
```

### 7.4 常量默认值定义

```cpp
// ✅ 正确：在 print_constant.h 中统一定义默认值
constexpr uint32_t NUMBER_UP_DEFAULT_VALUE = 1;
constexpr uint32_t NUMBER_UP_LAYOUT_DEFAULT_VALUE = NUMBER_UP_LAYOUT_LRTB;
constexpr uint32_t MIRROR_DEFAULT_VALUE = PRINT_MIRROR_DISABLED;
constexpr uint32_t PAGE_BORDER_DEFAULT_VALUE = PRINT_PAGE_BORDER_NONE;

// ❌ 错误：默认值散落在各处，没有统一管理
const int DEFAULT_NUMBER_UP = 1; // 某处
#define DEFAULT_LAYOUT 0         // 另一处
```

## 9. 总结：高效使用 AI 的关键

| 原则 | 说明 |
|------|------|
| **上下文要完整** | 告诉 AI 在哪个文件、参考什么、什么规范 |
| **分步骤进行** | 不要一次性大段生成，分步检查 |
| **让 AI 解释** | 确保你理解 AI 写的代码 |
| **人工作最后把关** | AI 是助手，决策和 review 必须在人 |
| **明确约束条件** | CUPS 只支持特定值这类约束必须告知 AI |
| **不要粘贴敏感信息** | 脱敏后描述，或只描述数据特征 |

---

**核心一句话**：AI 辅助开发的效率取决于你描述需求的质量。把 AI 当作一个聪明但不了解你项目的同事，背景交代越清楚，帮助越大。
